# モデル選定根拠 — cocoro-llm-server

## Primary: Llama 4 Scout 109B Q4_K_M

### 選定理由

| 評価項目 | 評価 | 詳細 |
|---|---|---|
| 推論品質 | ◎ | Llama 4世代・MoE構造で109B相当の品質 |
| コンテキスト長 | ◎ | 最大10Mトークン（実用は32K） |
| 日本語性能 | ○ | 多言語学習済み |
| VRAM効率 | ○ | Q4_K_M量子化で55GB、品質劣化最小 |
| vLLM対応 | ○ | AWQ/GGUF量子化をvLLMがネイティブサポート |
| ライセンス | ◎ | Meta Llama 4 Community License（社内利用OK） |

### Q4_K_M を選んだ理由

- Q4_K_M: 中間層に K量子化を適用した4bit量子化
- FP16（フル精度）比で **品質劣化 1〜2%** 程度で **VRAM 55%削減**
- Q5_K_M と比較して品質差は軽微だが VRAM 節約が大きい
- 96GB VRAM で2モデル同時起動を実現するための**必須の選択**

## Secondary: Qwen 3.5 32B Q5_K_M

### 選定理由

| 評価項目 | 評価 | 詳細 |
|---|---|---|
| 日本語性能 | ◎ | Alibaba製・日本語チューニング優秀 |
| 推論速度 | ◎ | 32Bサイズで高速応答 |
| 指示追従 | ◎ | Instruct版で指示精度が高い |
| VRAM | ○ | Q5_K_M量子化で22GB |
| ライセンス | ◎ | Apache 2.0（商用利用OK） |

### Q5_K_M を選んだ理由

- Q5_K_M: 5bit量子化でFP16比 **品質劣化 0.5%未満**
- 32Bモデルはパラメータが少ないため量子化劣化が出やすい
- Q4_K_Mでは日本語の微妙なニュアンスが失われるケースがある
- 22GBは十分なVRAM余裕（Primary 55GB + Secondary 22GB + KV 10GB = 87GB < 96GB）

## 代替候補と不採用理由

| モデル | 不採用理由 |
|---|---|
| Llama 3.3 70B | Llama 4 Scoutより品質・コンテキスト長で劣る |
| Mistral Large 2 | ライセンス制限あり（商用利用要申請） |
| Gemma 3 27B | 日本語性能がQwen 3.5より劣る |
| DeepSeek R1 70B | 蒸留版では推論品質が不安定 |
| Qwen 3.5 72B | VRAMが40GB必要でPrimaryと同時起動不可 |

## VRAM最適化の判断プロセス

```
96GB 総VRAM
  ├── Primary (Scout 109B): できるだけ大きいモデルかつ最高品質
  │   → Q4_K_M が55GBに収まる最大モデル → 採用
  │
  ├── Secondary (Qwen 32B): 24GB以下で高品質な日本語モデル
  │   → Q5_K_M で22GB → Qwen 3.5 32B 採用
  │
  ├── KVキャッシュ: 同時セッション数に依存
  │   → 10GB で5〜10セッション対応
  │
  └── 予備: LoRA実験・将来の拡張用
      → 9GB 確保
```
